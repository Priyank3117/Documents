A developer's mental model

Claude doesn't have one memory.
It has four.

The single biggest misunderstanding is imagining one big brain. In reality Claude works across four distinct storage layers — each with its own purpose, lifetime, and answer to the question "will this survive?" Think of it the way a software project spreads across your account, your repo, your terminal, and your RAM.

User Memory · permanent Project Knowledge · persistent Conversation · ephemeral Context Window · volatile

01 The pipeline

How Claude assembles a single answer

Every response is built by pulling from the durable layers, then loading whatever is most relevant into a working space to reason over. Read it top to bottom:

Your long-term preferences, if the feature is available on your account.

4

The Context Window

As much relevant information from the layers above as physically fits.

✓

Claude's answer

Generated from whatever was loaded into the window.

The key correction. The Context Window is not another memory source. It is the workspace (RAM) where Claude loads information from the other three. It doesn't store anything itself — it's the space where information is temporarily available while Claude reasons.

A cleaner one-line summary: Claude gathers information from User Memory, Project Knowledge, and the current conversation, then loads as much relevant information as fits into the Context Window to generate an answer.

02 The storage stack

Four layers, ordered by how long they last

The colours aren't decoration — they run along a persistence spectrum. Cool means permanent; warm means volatile. Hover any layer to inspect it.

1 · User Memory ≈ your GitHub profile lifetime: months → years

Facts about you, not your project: preferred language, coding style, formatting habits, favourite framework, how you want Claude to answer. It's your account — repositories come and go, but the account always exists. Stays until you edit it, remove it, or ask Claude to forget. Belongs here: "I write Playwright tests in C#," "keep answers concise," "SQL first." Not here: bug numbers, current sprint, this branch's architecture.

Persistence
Survives new chats & new computers
2 · Project Knowledge ≈ your Git repository lifetime: as long as the files exist

Everything about the current repository: architecture, design docs, coding guidelines, requirements, README, and files like memory.md or CLAUDE.md. Claude reads these like any documentation. Different branches can carry different docs — switch branches and Claude sees different files without touching User Memory. Its durability depends entirely on where the files live (see layer's persistence below).

Persistence
Safe if committed / backed up — gone if only local
3 · Conversation Context ≈ your terminal session lifetime: until the chat ends

Today's chat. Ask Claude to "explain Playwright" and later "show an example" — it remembers because it's inside this conversation, not User Memory. Like terminal history: close the terminal and it's gone. A brand-new chat starts fresh and does not automatically reuse the old one as context.

Persistence
Does not carry into a new chat
4 · Context Window ≈ your computer's RAM lifetime: this response only

The hardest one — because it isn't memory at all, it's RAM. A fixed budget (say 200K tokens) into which Claude loads conversation, User Memory, project files, the current prompt and code. Everything must fit. When there's too much, the least relevant material is dropped, exactly like RAM under pressure. Git stores everything; RAM only holds what's needed right now.

Persistence
Nothing is stored here
PERMANENT VOLATILE Every answer combines all four layers at once.

03 Sorting your information

User Memory vs. memory.md — what goes where

The most common mix-up: dumping project details into personal memory. Preferences are about you; project facts are about the repo.

✓ Put in User Memory

  • Preferred language & coding style
  • Formatting preference (e.g. concise, no emojis)
  • Favourite framework — "I use Playwright"
  • How Claude should answer — "SQL first, then explain"
  • Career role & writing style

✕ Keep out of User Memory

  • Bug 3156 / the current sprint
  • The branch you're on right now
  • Meeting notes & temporary tasks
  • This project's architecture
  • Anything that changes per branch

✓ Put in memory.md / project files

  • Repository & folder structure
  • Architecture and design docs
  • Business rules & requirements
  • Naming & testing conventions
  • Known issues, common mistakes, key files

What memory.md actually is

A README, but written for Claude. It is an ordinary project document — not Claude's built-in User Memory. Claude reads it when it's in the project or context; delete it and only that project knowledge disappears. Your personal User Memory is untouched.

04 The file question

Can you open or edit User Memory as a file?

Short answer — no. On Claude.ai it lives on Anthropic's servers, like your GitHub account settings: you change them through the interface, but you never get a file on your PC.

NOIs there a local file like ~/.claude/user_memory.md?

There's no local file, no memory.db, and no Git repo or cloud folder holding it. You can't download a memory.json.

NOCan you create User Memory by writing a file?

Dropping a user_memory.md into a folder won't auto-import. It stays an ordinary document unless you explicitly bring it into the conversation.

YESCan you view or update it through Claude?

Yes — conversationally. "Remember that I prefer C#," "Forget that I use Selenium," or "What do you remember about me?" is how you read and write it (where the feature is available on your plan).

YESCan you roll your own memory file instead?

Absolutely — a preferences.md you paste in at the start of a chat. It's version-controlled, editable anywhere, and portable across tools. But Claude won't recall it in future chats unless you provide it again or save the equivalent via built-in memory.

MOVING PC?What comes with you?

Log into the same account and your built-in User Memory travels with the account (not copied from the old PC). Local files like preferences.md only follow if you commit, copy, or cloud-sync them.

05 Persistence in practice

Local vs. committed — the survival test

Project knowledge isn't "permanent inside Claude." It's persistent only because something else stores it. Git is what makes it survive.

Local only

# lives only on your machine Project/ ├── README.md ├── memory.md ← in .gitignore └── docs/

PC crashes, disk fails, or you delete it — and it's gone. Teammates never had it. Exactly like deleting a README.

Committed & pushed

# backed by the remote git commit -m "docs" git push # later, on any machine… git clone ✓ everything restored

New laptop, fresh clone, and every doc comes back. Everyone on the team shares identical project knowledge.

Recommended: commit memory.md when the whole team should share it — treat it like README.md. Keep it local (or gitignored) only when it's your private notes. Best committed content: architecture, guidelines, folder structure, coding conventions.

07 The simple rule

Unsure where something belongs? Ask one question.

ASK YOURSELF →

"Do I want Claude to remember this across all future conversations?"

→ User Memory
ASK YOURSELF →

"Does everyone working on this repository need it?"

→ Project Knowledge (commit it to share)
ASK YOURSELF →

"Is it only relevant to this discussion?"

→ Conversation Context
ASK YOURSELF →

"Is it just information Claude is actively processing right now?"

→ Context Window

Best practice — use both systems together. Built-in User Memory for long-term personal preferences ("I write Playwright tests in C#"), and project files like CLAUDE.md / memory.md / README.md for repo-specific rules, architecture, and conventions. Keeping the two separate is what keeps everything maintainable over time.
