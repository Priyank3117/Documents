1. Why worktrees — the problem they solve

You wanted to work on two topics at the same time (e.g. group line items in one chat, another topic in another chat). The instinct is to do both on the same branch in the same folder, then stash one, PR the other, make a new branch, pop the stash, and PR again.

❌ Why "same branch, same folder, two chats" breaks A single folder has one working tree and one git index. Two sessions editing it at once cause:
  • index.lock contention (git blocks the second operation — we already hit a stale lock earlier today).
  • Chat B's uncommitted edits appear inside Chat A's diff — the two topics get tangled.
  • The stash → pop juggling can conflict and land the wrong files in the wrong PR.
✅ The fix Give each topic its own folder + own branch from the start using a git worktree. Two chats then run fully in parallel, and each produces its own clean PR — no stashing at all.

2. What a worktree actually is

Normally: one folder → one working tree → only one branch checked out at a time. To see another branch you git checkout, which swaps the files in place — that's why you can't view two branches at once.

A worktree is an extra working folder attached to the same .git database.

┌───────────────────────────────────┐ │ ONE shared .git database │ │ (all commits, branches, remotes) │ └───────────────────────────────────┘ ▲ ▲ ▲ ┌────────────┘ │ └────────────┐ MAIN folder worktree #1 worktree #2 ...Stabilize.../ ../group_line_items/ ../other_topic/ branch: main branch: group_line_items branch: other_topic (own files + HEAD) (own files + HEAD) (own files + HEAD)
Shared across all worktreesSeparate per worktree
The single .git database: every commit, branch, remote, fetch and push is visible everywhere. Its own files on disk, its own staging area (index), and its own checked-out branch (HEAD).
🔒 The rule that protects you The same branch cannot be checked out in two worktrees at once — git refuses. This is a feature: it makes collisions impossible.

3. The command plan — in exact order

All run from your main repo folder unless noted.

#CommandWhat it does
1git fetch originUpdates the remote state (origin/main) so new branches start from the newest main.
2git worktree add ../worktrees/group_line_items -b AnMehta/group_line_items origin/mainCreates folder + new branch #1, checked out there.
3git worktree add ../worktrees/other_topic -b AnMehta/other_topic origin/mainCreates folder + new branch #2.
4git worktree listShows every worktree and the branch each is on (verification).

4. Anatomy of git worktree add

git worktree add ../worktrees/group_line_items -b AnMehta/group_line_items origin/main │ │ │ │ the action WHERE (new folder, NEW branch name STARTING point outside main folder) (base it on remote main)

What git physically does: it copies the checked-out files of the target branch into the new folder and writes small linking metadata. It does not duplicate the history — that stays shared — so a worktree is much lighter than a full clone.

5. The per-worktree work cycle

Inside each worktree folder this is ordinary git — nothing special:

cd ../worktrees/group_line_items          # this folder IS the group_line_items branch
# ...edit files...
git add .
git commit -m "meaningful message"        # automatically lands on THIS folder's branch
git push -u origin AnMehta/group_line_items   # first push (-u sets upstream)
git push                                  # every push after that
# ...raise the PR in the Azure DevOps web UI...
git worktree remove ../worktrees/group_line_items   # cleanup after the PR is merged
You never run git checkout here. Each folder is permanently tied to its own branch. The folder is the branch.

6. Opening the two worktrees in separate VS Code windows

Option A Command line (fastest):

code ../worktrees/group_line_items     # opens window #1
code ../worktrees/other_topic          # opens window #2

(The code command works if VS Code added it to PATH — Command Palette → "Shell Command: Install 'code' command in PATH".)

Option B Inside VS Code (no CLI):
File → New Window → Open Folder… → pick the folder. Repeat for the second.

┌────────── VS Code Window 1 ──────────┐ ┌────────── VS Code Window 2 ──────────┐ │ Folder: group_line_items │ │ Folder: other_topic │ │ Branch: AnMehta/group_line_items │ │ Branch: AnMehta/other_topic │ │ (edit group line-item tests here) │ │ (edit the other topic here) │ └──────────────────────────────────────┘ └──────────────────────────────────────┘ └────────── both share the same .git history ──────────┘
Why editing one can't affect the other They are two different folders holding two physical copies of the files. An edit in window 1 touches only the files in folder 1. They share only the invisible .git history — never the working files — so nothing crosses over until you commit, push, and merge. Each window's bottom-left status bar permanently shows its own branch.

7. How many worktrees can you make?

  • Git has no fixed limit — you can make dozens.
  • Real limits are disk space (each worktree copies the working files) and your machine (each running build/test uses CPU + RAM).
  • The one hard rule: the same branch can only be checked out in one worktree at a time.
  • Your original main folder counts as worktree #1, so after the two commands you have 3 total.
  • Practical advice: keep 2–4 at once, or it gets hard to track.

8. Pushing a specific worktree

⚠️ First, the key fact The two worktrees are on different branches — they never share one. So pushing/merging/deleting one never disturbs the other.

Can I push only one worktree? Yes — because each is its own branch, pushing one pushes only that branch.

Which command? Git pushes a branch, not a folder. Two equivalent ways:

Option A From inside that worktree's folder (pushes that folder's current branch):

cd ../worktrees/group_line_items
git push -u origin AnMehta/group_line_items   # first push
git push                                       # thereafter

Option B From any folder, name the branch explicitly:

git push origin AnMehta/other_topic
  • git push with no arguments pushes the current branch of the folder you're standing in — that's the "push just this worktree" behaviour.
  • -u (--set-upstream) is only needed on the first push of a new branch.

9. Deleting the branch after a PR is merged

Should you avoid deleting the branch on Azure DevOps? No. Deleting one branch after its PR merges has zero effect on the other branch or its worktree — they're independent. The normal habit (delete on merge, or let Azure auto-delete) is perfectly safe.

Just follow the correct local cleanup order afterwards:

# 1) remove the worktree folder FIRST
#    (git won't delete a branch that a worktree still occupies)
git worktree remove ../worktrees/group_line_items

# 2) optional: delete the now-merged local branch
git branch -d AnMehta/group_line_items

Deleting the remote branch (Azure) and cleaning up locally are independent — do both, one branch at a time.

10. Where the folders physically live

The folder is created exactly where the path points, relative to where you run the command. ../ means "one level up" (a sibling of your repo).

Your repo:

C:\Users\AnMehta\SpecFlowTest\CPQPlatformTests_CSharp_Stabilize_test_pipeline

So git worktree add ../worktrees/group_line_items ... creates:

C:\Users\AnMehta\SpecFlowTest\worktrees\group_line_items

Resulting layout on disk

C:\Users\AnMehta\SpecFlowTest\ ├── CPQPlatformTests_CSharp_Stabilize_test_pipeline\ ← main repo (holds the real .git\ folder) │ └── .git\ │ └── worktrees\ ← tiny metadata for each extra worktree │ ├── group_line_items\ │ └── other_topic\ └── worktrees\ ├── group_line_items\ ← worktree #1: full copy of files + a .git FILE (not folder) └── other_topic\ ← worktree #2: full copy of files + a .git FILE (not folder)

The technical detail

  • A worktree folder has all your project files — a real working copy you can build and edit.
  • Instead of a .git folder, it holds a small .git file — one line pointing back to the main repo:
gitdir: C:/Users/AnMehta/SpecFlowTest/CPQPlatformTests_CSharp_Stabilize_test_pipeline/.git/worktrees/group_line_items

That's why worktrees are cheaper than cloning — they borrow the one shared .git database and only duplicate the working files.

11. Best-practice placement

✅ Do
  • Put worktrees outside the main repo folder — as siblings or in a dedicated parent.
  • Prefer a dedicated worktrees\ parent folder so everything stays grouped and tidy.
  • Give folders descriptive names that match the branch/topic.
  • Keep them on the same drive (C:) — simplest and fastest.
❌ Don't
  • Never create a worktree inside the main repo folder (e.g. ./other_topic) — git would try to track it and tools would recurse into it.
  • Don't move a worktree by dragging in Explorer — use git worktree move so the link stays intact.

Two layout options

Option 1 — flat siblings (simplest):

SpecFlowTest\CPQPlatformTests_CSharp_Stabilize_test_pipeline\   (main)
SpecFlowTest\group_line_items\
SpecFlowTest\other_topic\

Option 2 — dedicated worktrees\ parent (recommended):

git worktree add ../worktrees/group_line_items -b AnMehta/group_line_items origin/main
git worktree add ../worktrees/other_topic      -b AnMehta/other_topic      origin/main
SpecFlowTest\CPQPlatformTests_CSharp_Stabilize_test_pipeline\   (main)
SpecFlowTest\worktrees\group_line_items\
SpecFlowTest\worktrees\other_topic\
If you ever lose track git worktree list prints the absolute path and branch of every worktree — your single source of truth.

12. Plain-English analogy

Think of your .git folder as a single filing cabinet that stores every version of every document your team ever saved.

  • Normally you have one desk in front of it and can lay out only one project's papers at a time. Switching projects means clearing the desk (git checkout).
  • A worktree is a second desk pulled up to the same cabinet. Each desk has its own papers spread out — Desk A = "group line items", Desk B = "other topic". Each stays exactly as you left it.
  • VS Code windows are your eyes on each desk. Scribbling on Desk A can't smudge Desk B — separate stacks of paper.
  • When a job is done you file its papers back (commit + push), hand it in for review (the PR), then fold up that desk (git worktree remove).
  • The cabinet is shared, so once papers are filed, every desk can see them.

The big win: no more hiding papers in a drawer to work on the other set (that's the stash/pop juggling) — you simply keep a separate desk per job.

13. One-page cheat-sheet

# ── SET UP (from the main repo folder) ───────────────────────────
git fetch origin
git worktree add ../worktrees/group_line_items -b AnMehta/group_line_items origin/main
git worktree add ../worktrees/other_topic      -b AnMehta/other_topic      origin/main
git worktree list

# ── OPEN EACH IN ITS OWN VS CODE WINDOW ──────────────────────────
code ../worktrees/group_line_items
code ../worktrees/other_topic

# ── WORK CYCLE (inside a worktree folder) ────────────────────────
git add .
git commit -m "message"
git push -u origin AnMehta/group_line_items    # first push only
git push                                        # thereafter
#   ...then raise the PR in Azure DevOps...

# ── PUSH A SPECIFIC BRANCH FROM ANYWHERE ─────────────────────────
git push origin AnMehta/other_topic

# ── CLEANUP AFTER A PR MERGES ────────────────────────────────────
git worktree remove ../worktrees/group_line_items   # remove folder FIRST
git branch -d AnMehta/group_line_items              # then delete local branch

# ── HANDY ────────────────────────────────────────────────────────
git worktree list         # where is each worktree + its branch
git worktree move OLD NEW  # relocate safely (never drag in Explorer)
Golden rule One worktree = one folder = one branch = one PR. They never share a branch, so pushing, merging, and deleting one never disturbs the other. Keep worktree folders outside the main repo, and put them nested inside a repo folder never.
