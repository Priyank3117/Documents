Why azure-pipelines-sanity.yml shouldn't exist as a near-duplicate of azure-pipelines.yml — and how a single file can drive the sanity run, the regression run, and manual ad-hoc runs on its own.

01 The problem

Two files that are 99% identical will drift apart

Right now azure-pipelines-sanity.yml is effectively a byte-for-byte copy of azure-pipelines.yml. The only real differences are the schedules: block and the default TestFilter (plus a couple of cosmetic displayName tweaks).

The risk isn't today — it's every change after today. A new step, a task version bump, a filter tweak: each one has to be manually mirrored into both files, or they quietly fall out of sync. That silent divergence is configuration drift.

02 The moving parts

Five terms that make the single-file design click

Pipeline

A registration in Azure DevOps that knows when to start and which YAML to run. It holds no steps itself — it just triggers execution (commit, PR, manual, or schedule).

YAML file

The actual instruction manual: variables, jobs, steps, conditions, and schedules. The pipeline just points at it and follows it.

Schedule

Only a timer. A cron entry says "start the pipeline at this time." It knows nothing about smoke, sanity, or regression — it just fires.

displayName (on a schedule)

A human label on a cron entry, e.g. "Daily Sanity". It's more than cosmetic — Azure exposes it at runtime.

Build.CronSchedule.DisplayName

The predefined variable Azure sets to the displayName of whichever schedule fired the run. That single value is how one YAML file can tell which timer woke it up and branch accordingly — no second file required.

04 The mechanism in code

A named schedule, plus a compile-time filter switch

Add the sanity timer alongside the existing schedules, then let a template expression choose the filter based on which schedule fired (or the manual parameter). Both live in the one existing azure-pipelines.yml.

azure-pipelines.ymlschedules
schedules:
- cron: "30 19 * * *"
  displayName: "Daily Sanity run at 1:00 AM IST"
  branches:
    include: [main]
  always: true
# (plus the existing full-regression schedule here too,
#  if it isn't already a classic UI trigger)
azure-pipelines.ymlvariables
variables:
  ${{ if eq(variables['Build.CronSchedule.DisplayName'], 'Daily Sanity run at 1:00 AM IST') }}:
    effectiveTestFilter: 'TestCategory=sanity&FullyQualifiedName!~CustomerTests&FullyQualifiedName!~Salesforce'
  ${{ elif and(ne(parameters.TestFilter, ''), ne(parameters.TestFilter, 'all')) }}:
    effectiveTestFilter: ${{ parameters.TestFilter }}&FullyQualifiedName!~CustomerTests&FullyQualifiedName!~Salesforce
  ${{ else }}:
    effectiveTestFilter: 'FullyQualifiedName!~CustomerTests&FullyQualifiedName!~Salesforce'

Read top-down: if the sanity schedule fired, run sanity. Otherwise, if a real manual TestFilter was passed, honour it. Otherwise fall through to the full regression default. One file covers all three paths.

05 When keeping them separate is fine

Not a blocker — just the alternative worth considering

Legitimate reasons to keep a second pipeline

Consolidation is about the YAML duplication. There are real operational reasons a team might still want two separate pipeline registrations:

  • Independently enable / disable one without touching the other
  • Separate permissions or approvals
  • Separate retention policies
  • Cleaner, separated run history
  • Ability to delete one without affecting the other

If none of those apply, one pipeline with multiple named schedules is simpler and drift-proof. And note: even two pipelines could point at the same YAML — but that's a different pattern than the one recommended here.

The recommendation is one file, one pipeline registration — not two pipelines sharing a file, and not two near-identical files to keep in sync.
