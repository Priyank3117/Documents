Designer's mind · the one idea that explains everything

If you keep one thing: a test is a pipeline that assembles state, then asserts on it — and the architecture is organised around who owns which state and for how long. Config is run-lived; the browser and the ScenarioContextProperties bag are scenario-lived; a page object is view-lived. Every decision — DI restraint, hooks, the page factory, cleanup, parallelism — falls out of managing those three lifetimes safely under 10 parallel workers.

The mental model — how execution & data flow

Designer's mind · why steps live with pages, not features

Step definitions are not in the test project — they sit in PageObjects next to the pages they drive, wired via reqnroll.json → bindingAssemblies. A step is glue between Gherkin and a page object, so it belongs beside the page, and the pair ships as one reusable library. This is what lets TestCustomObjects be a second binding assembly overriding behaviour per customer without touching core.

The seven design qualities, and how each is achieved

An analogy: think of the framework as a factory line. Config is the loading dock (materials in), the API layer stamps raw parts (test data), page objects are the assembly stations, steps are the work orders, and reporting is quality inspection at the end. Each station does one job and hands off cleanly — which is exactly what these qualities describe:

Architecture

Five assemblies in strict dependency order; the domain DLL at the centre.

Layering

Gherkin → steps → pages/services → Playwright/HTTP → app. No layer skips another.

Separation of concerns

UI mechanics on pages, transport in services, lifecycle in hooks, state in the context bag.

Maintainability

A DOM change touches one Selectors class; an endpoint change touches one service.

Scalability

10 parallel workers via per-scenario isolation + per-worker profiles; add capacity by adding profiles.

Testability

API-first setup + fail-soft cleanup make each scenario independently runnable and repeatable.

Reusability

Shared components, grid factories, and the ApiService base are written once and reused everywhere.

Every package explained

For each NuGet: why it exists, the problem it solves, alternatives, why chosen, its key namespace/classes/methods used here, and how hard it'd be to build yourself.

Reqnroll 3.3.0 (+ Reqnroll.NUnit)

Problem it solves
Turns Gherkin (.feature) into runnable tests and routes each step line to a C# method. It is the SpecFlow successor (SpecFlow is EOL) — the choice was essentially forced by that migration.
Alternatives
SpecFlow (dead), raw NUnit/xUnit (no Gherkin), Cucumber (JVM/JS), LightBDD (fluent, not Gherkin-file based).
Key namespaces/classes
Reqnroll: [Binding], [Given/When/Then], ScenarioContext, FeatureContext, Table, [StepArgumentTransformation], [Scope]; Reqnroll.BoDi.IObjectContainer (DI); ReqnrollOutputHelper (attachments).
Build it yourself?
Very hard — you'd re-implement a Gherkin parser, a source generator that emits NUnit tests, a regex-based binding registry, and a DI container. This is the package you never rebuild.
Docs
docs.reqnroll.net — Bindings, Hooks, Context Injection, Configuration.

Microsoft.Playwright 1.52.0

Problem
Cross-browser automation with auto-waiting, network control, tracing, and isolated browser contexts.
Alternatives
Selenium WebDriver (older, flakier waits, no built-in trace), Cypress (JS-only), Puppeteer (Chromium-only).
Why chosen
Auto-waiting locators remove most explicit sleeps; BrowserContext gives cheap per-test isolation (crucial for 10× parallel); Trace Viewer + video for debugging.
Key classes/methods used
Playwright.CreateAsync, IBrowser.NewContextAsync, IPage.Locator/GotoAsync/ScreenshotAsync, ILocator.ClickAsync/Filter, WaitForLoadStateAsync, Tracing.Start/StopAsync.
Build it yourself?
Impractical — it drives browsers over the DevTools/WebKit protocols.
Docs
playwright.dev/dotnet — Locators, Auto-waiting, Browser Contexts, Trace Viewer.

NUnit 3 + Microsoft.NET.Test.Sdk 18 + NUnit3TestAdapter

Problem
The actual test runner underneath Reqnroll — discovers tests, runs them in parallel, reports results to dotnet test / VSTest.
Why NUnit
Its parallelism model ([Parallelizable(ParallelScope.Fixtures)], [LevelOfParallelism], [NonParallelizable]) is the most mature — and it's what reqnroll.json's unitTestProvider targets.
Key APIs used
TestContext.AddTestAttachment (evidence), the assembly-level parallelism attributes, Assert.Warn (fail-soft cleanup).
Docs
docs.nunit.org — Attributes → Parallelizable; framework-parallel-tests.

Microsoft.Extensions.Configuration (+ .Json, .Binder, .EnvironmentVariables)

Problem

Alternatives

ConfigurationBuilder, AddJsonFile, AddEnvironmentVariables, IConfiguration.Bind.

Easy for one file; hard to match provider ordering + binding cleanly. Use the platform.

Section 06

Configuration layer

One job: turn flat JSON + env vars into a resolved, per-worker target URL and credentials — without leaking CI secrets into code.

Purpose

Centralise all environment-dependent values behind one loader.

Problem

Same tests, many envs (dev/test/stage) and orgs, with CI overriding without a rebuild.

Pattern / SOLID

Layered provider + lazy cache + typed binding · SRP (config isolated from tests).

Packages / .NET

Microsoft.Extensions.Configuration.* · C# 13 field keyword; expression-bodied members.

// SpecFlowTests/TestInfrastructure/AppConfiguration.cs
public static IConfiguration AppSettingsConfiguration =>
    field ??= GetAppConfiguration("appsettings.json");     // lazy static cache

private static IConfiguration GetAppConfiguration(string configName) =>
    new ConfigurationBuilder()
        .SetBasePath(Directory.GetCurrentDirectory())
        .AddJsonFile(configName)
        .AddEnvironmentVariables()      // ← last provider wins → CI overrides JSON
        .Build();
Designer's mind · why env vars come last

Provider order = precedence. JSON gives local defaults; the env layer lets Azure's FileTransform/variables override RunMode, Headless, Xray.* at runtime. Straight from the MS recipe — use the platform's config system, don't rebuild it.

Execution & data flow: URL resolution

 RunMode="stage"        org.Url="autotestspecflowui.vendavocpq.com"
      │                       │
      ▼  UrlHelper.BuildUrl:  "cpqstage-" + org.Url
      ▼
 org.FullUrl = "https://cpqstage-autotestspecflowui.vendavocpq.com"
      ▼  (OrganizationResolver sets it; allocates a per-worker Profile)
 Playwright BaseURL + REST base + login credentials

UrlHelper maps RunMode→a Version enum→prefix; OrgHelper.GetOrganization binds the Organizations section to List<Organization> (case-insensitive match, throws if missing); OrganizationResolver combines them and allocates a distinct Profile per worker for parallel safety.

How to extend · common mistakes · exercise

  • Extend: new setting → add to appsettings.json, read via AppConfiguration.AppSettingsConfiguration["Key"] or bind a POCO.
  • Mistake: reading Directory.GetCurrentDirectory()-relative files that aren't copied to output — the .csproj must CopyToOutputDirectory them (it does for both JSON files).
  • Exercise: add a SlowMoOverride key and thread it into BrowserSetupConfiguration without touching any step.
Watch-out · secrets in the repo

Real Xray secret, Jira token, and org passwords are committed in plaintext. The env-var layer already supports overrides — move secrets to pipeline secret variables / a key vault and delete them from tracked files. Highest-priority hygiene fix.

Section 07

Dependency injection (BoDi)

Reqnroll ships a tiny IoC container — BoDi. This framework uses it deliberately sparingly; understanding where it stops is the lesson.

// TestHooks.RegisterDependencies — the ONLY registration site
container.RegisterInstanceAs(playwright);   // IPlaywright
container.RegisterInstanceAs(browser);      // IBrowser
container.RegisterInstanceAs(page);         // IPage
context.Set(page, ContextKeys.BrowserPage);
context.Set(new LoginPage(page), ContextKeys.CurrentPage);   // first view = login
context.Add(ContextKeys.BaseUrl, organization.FullUrl);
context.Add(ContextKeys.BrowserContext, browserContext);
Designer's mind · why page objects are NOT in the container

A page object represents "the screen you're on" — runtime state, not a static dependency. You can't know at registration time whether the user is on Basket or Quote. So pages stay out of DI; the current page lives in ScenarioContext under CurrentPage, swapped by a factory (UpdateCurrentPage()) after each navigation. DI holds what lives for the scenario (browser); the context holds what changes (view). Two lifetimes, two mechanisms.

How steps receive dependencies
Constructor injection only: public sealed class BasketPageSteps(ScenarioContext ctx) : StepsBase(ctx). Reqnroll resolves ScenarioContext (and hook params like IObjectContainer, IPage, ReqnrollOutputHelper) from BoDi.
Alternatives
Microsoft.Extensions.DependencyInjection via a Reqnroll plugin (heavier), or manual singletons (loses scenario scoping). BoDi is right-sized for scenario lifetime.
Why inheritance here (StepsBase)
Every step needs the same typed context accessors — pushing them to a base class is DRY and gives one place to change context access. Composition would mean each step holds a helper field; inheritance is lighter for this ubiquitous, uniform need.
Section 08

Test lifecycle — the hooks

TestHooks.cs is the conductor. Four scopes (run / scenario / step) where every cross-cutting concern is bolted onto the test.

[BeforeTestRun] (once, static)  load config ─► OrganizationResolver.Init ─► delete artifacts ─► (opt) create Xray execution
   ┌──────────────── per scenario ×10 parallel ────────────────┐

   │   context.Add(ScenarioContextProperties)                   │

   │      → collect CucumberStep{keyword,text,status,dur,error}  │

Fig 3 · Reqnroll creates one TestHooks instance per scenario — static fields hold run state, instance fields hold scenario state. That's why it's parallel-safe.

Designer's mind · reporting is a side effect, never a gate

Every Xray/Jira call is wrapped in try/catch so a flaky reporting API can't fail a passing test. Counters use Interlocked because ten workers report concurrently. Observability must degrade silently, not take the run down.

Good practice · deterministic teardown

Browser created in BeforeScenario, disposed in AfterScenario (browser.CloseAsync() → playwright.Dispose()). One full stack per scenario = zero shared browser state. Isolation by construction beats isolation by discipline.

Section 09

Browser lifecycle

All Playwright creation is centralised in BrowserSetupConfiguration.cs so no page or step ever launches a browser.

var playwright     = await Playwright.CreateAsync();
var browser        = await LaunchBrowserAsync(playwright, launchOptions);  // chromium/firefox/webkit switch
var browserContext = await browser.NewContextAsync(contextOptions);        // BaseURL, ReducedMotion, video
ConfigureBrowserTimeouts(browserContext);
var page           = await browserContext.NewPageAsync();
await SetViewPortIfNeededAsync(page);
return (playwright, browser, browserContext, page);
Designer's mind · the settle helper is the framework's heartbeat

PlaywrightExtensions.WaitForLoadingAsync() runs after almost every click: waits NetworkIdle+DOMContentLoaded+Load, then for spinners (.fa-cog) and modal backdrops (.modal-backdrop.fade) to hide, then body.modal-open to detach — all in try/catch{}. It swallows errors because a settle helper that throws turns every timing blip into a failure; its job is to wait as much as possible, not to assert. The most important 20 lines in the framework, and why the suite survives a flaky SPA at 10× parallel.

Extension methods, why: ClickAndWaitForLoadingAsync/WaitForLoadingAsync are extension methods on IPage — you can't modify Playwright's sealed types, and an extension reads like a native call (page.WaitForLoadingAsync()) while living in your code. That's exactly when extension methods are the right tool: adding verbs to a type you don't own.

Section 11

How Reqnroll executes a scenario

How does one Gherkin line find the one C# method that runs it? Here's the machinery — compile time and run time.

 BUILD TIME
   *.feature ──► Reqnroll generator ──► *.feature.cs  (one NUnit [Test] per scenario)
                 reads reqnroll.json: unitTestProvider=NUnit, bindingAssemblies
 RUN TIME
   NUnit runs [Test] ──► Reqnroll runtime
        │ builds the binding registry ONCE: reflect over bindingAssemblies,
        │ find [Binding] classes, index every [Given]/[When]/[Then] regex
        ▼ per step line:
        match text vs indexed regexes ──► exactly one?  0→"no match"  2+→"ambiguous"
        resolve binding class from BoDi (ctor gets ScenarioContext)
        bind captured groups + run transformations ──► method params
        invoke async method ──► pass/fail

Fig 4 · The generator makes NUnit tests at build; the runtime does regex routing at execution. Reflection runs once to build the registry, not per step.

Designer's mind · "why does only the current step run?"

Reqnroll doesn't scan all definitions each line. At startup it builds an in-memory index (regex → method) from the binding assemblies; each step is matched against that index and the runtime enforces exactly-one — zero or two-plus matches are hard errors. This is why PageName, prefixes + ^…$ anchoring + alternation groups matter: they keep every regex unambiguous. Duplicates are resolved by naming discipline and, for customers, [Scope]. Performance is fine because matching is a dictionary/regex scan over a fixed index, and reflection is amortised to once.

Section 13

Step definitions

Steps are glue and nothing more: translate a sentence into a page/service call and update context. No business logic, no locators, no substantive assertions in the step itself.

[Binding]
public sealed class ConfigureSteps(ScenarioContext scenarioContext) : StepsBase(scenarioContext)
{
    private ConfigurePage ConfigurePage => (ConfigurePage)CurrentPage;

    [When(@"Configure, I add the products to Quote")]
    [When(@"Configure, I add the products to Price Agreement")]   // 2 phrasings, 1 method
    public async Task WhenIAddTheProductsToBasket()
    {
        await ConfigurePage.AddToBasket();   // delegate to the page
        await UpdateCurrentPage();           // re-derive the view after navigation
    }
}
Why business logic is avoided here
A step is a translation layer. Logic in steps can't be reused by other steps and hides behind Gherkin, where it's untestable and undiscoverable. Behaviour belongs on the page (UI) or service (API); the step only orchestrates.
Why page objects are called (not raw Playwright)
So selectors and waits live in one place per screen — change the DOM once, not in every step.
Why API calls may appear in steps
Only in ApiSteps/, for data setup — and even then they delegate to a *Service, never issue raw HTTP.
Responsibilities that belong here
Cast CurrentPage, call one page/service method, refresh via UpdateCurrentPage(), occasionally branch on a returned modal type, store created entities.
Section 14

ScenarioContext — deep dive & debugging

The spine of the run. Where it's created, how long it lives, how it's injected, how state flows, how to debug it, and the mistakes to avoid.

What / why
A scenario-scoped key/value bag Reqnroll provides so independent step methods can share state (step 5 sees the quote created in step 1).
Where created / how long it lives / disposal
Reqnroll creates it before the first step and disposes it after the last — lifetime = exactly one scenario. Nothing survives into the next scenario (which is what isolation demands).
How injected
Via constructor injection into every [Binding] class (and hook methods) by BoDi.
Thread safety / parallel
Confined to one scenario ⇒ effectively single-threaded for ordinary reads/writes; no locking needed except genuine cross-thread touch-points (AddCreatedEntity lock, run-wide Interlocked counters).

This framework's refinement: one typed carrier, not many keys

 ScenarioContext (Reqnroll, scenario-scoped)  — keyed by ContextKeys constants (no magic strings)
   ├── BrowserPage            → IPage
   ├── BrowserContext         → IBrowserContext
   ├── CurrentPage            → PageObjectBase   (current view; swapped by the factory)
   ├── BaseUrl                → string
   └── ScenarioContextProperties → ONE typed bag holding everything else:
          Organization, CreatedEntities (drives cleanup), CurrentQuote, CurrentBasket,
          CurrentPriceAgreement, CurrentSmartCatalog, UserSession, UpdatedApplicationSettings ...
Designer's mind · typed bag beats loose Get/Set<T> everywhere

The naïve way sprays Set(quote,"quote")/Get<Quote>("quote") across steps — stringly-typed, un-discoverable, refactor-hostile. Instead one ScenarioContextProperties object is stored and its typed properties mutated: ScenarioContextProperties.CurrentQuote = quote;. Benefits: IntelliSense, compile-time safety, one place to see all scenario state, a natural home for CreatedEntities. All raw Get/Set is hidden in StepsBase (the Facade) — no step touches the context API directly.

// PageObjects/StepsBase.cs — the facade every step inherits
protected PageObjectBase CurrentPage {
    get => CurrentContext.Get<PageObjectBase>(ContextKeys.CurrentPage);
    set => CurrentContext.Set(value, ContextKeys.CurrentPage);
}
protected ScenarioContextProperties ScenarioContextProperties =>
    CurrentContext.Get<ScenarioContextProperties>(ContextKeys.ScenarioContextProperties);

How to debug / visualize what's stored

  • Inspect the bag: set a breakpoint in any step and expand ScenarioContextProperties — one object shows the entire scenario state (a direct payoff of the single-carrier design).
  • Dump on failure: in AfterScenario you can serialise ScenarioContextProperties to the attached output for post-mortem (the hook already has the context).
  • Placeholder not resolving? The value wasn't set — check the create-step actually assigned Current*. The transform reads what's there; it can't invent a value.
Common mistakes

① Using ScenarioContext.Current — a static that breaks under parallel; always inject. ② Expecting FeatureContext to carry data — it isn't used here; scenario-scoped only. ③ Forgetting AddCreatedEntity after a create — the entity then leaks past cleanup. ④ Storing raw values under ad-hoc string keys instead of a typed property on the bag.

Page Object Model & the Base Page

Pages encapsulate how to drive a screen so steps stay declarative. Two signature moves: self-validating constructors and fluent navigation returning the next page.

// PageObjects/PageObjectBase.cs — primary ctor takes the Playwright page
public partial class PageObjectBase(IPage page)
{
    public IPage CurrentPage { get; set; } = page;
    public SideMenuComponent SideMenu { get; } = new(page);   // composed onto EVERY page

    protected void VerifyPageObjectUrlAndState(string pageObjectName)
    {   // asserts URL matches the page — called from every concrete ctor
        CurrentPage.WaitForLoadingAsync().GetAwaiter().GetResult();
        CurrentPage.Url.Should().MatchRegex(/* from pageObjectName + CpqUrlPattern */);
    }
}
Why BasePage exists
Every page shares: a Playwright page reference, the side menu, URL/state verification, and the settle behaviour. Hoisting these to a base removes duplication and gives one place to evolve them.
What belongs on BasePage
Truly universal concerns: CurrentPage, SideMenu, GetScreenTitleAsync, VerifyPageObjectUrlAndState.
What must NEVER be on BasePage
Screen-specific selectors or actions (a ClickSubmitQuote belongs on BasketPage, not the base) — that would couple every page to one screen's DOM.
Inheritance vs composition here
Pages inherit BasePage (shared identity/behaviour) but compose components (SideMenu, tables) — inheritance for "is-a page," composition for "has-a widget."
Designer's mind · the self-validating constructor

Every concrete page calls VerifyPageObjectUrlAndState(nameof(BasketPage)) in its ctor. Consequence: you cannot hold a BasketPage unless you're genuinely on the basket screen. A navigation bug fails at construction, with a clear message, not ten steps later on a confusing "element not found." Type + runtime URL check together enforce "the object matches reality."

Shared components & the entity grid

The most sophisticated part of the framework: modals, tables, and editable cells modelled as composable objects that scope to any container and pick behaviour from the live DOM.

Page-scoped component

Subclass PageObjectBase, take IPage. Global widgets: SideMenuComponent, NavigationButtons.

Parent-scoped component

Take (IPage, string parentSelector) so the same widget works inside any container; every locator is prefixed with the parent selector.

 BasketBasePage.GetConfigurationTableComponent("Guitars")
   ─► EntityTableFactory ── table visible? ─► EntityTable (grid)   else ─► EntityTableStandard (cards)
   ─► new BasketTableComponent(page, table)     // TableComponent → Quotation → Basket chain
   ─► SetPropertyValueAsync(rowKey, column, value)
       ─► EntityTable → EntityRow → PropertyCellFactory (by data-datatype):
            RichText→RichTextPropertyCell(+modal) · Dropdown→DynamicDropdownPropertyCell(select2)
            Date→ValidityDatePropertyCell(+modal) · default→PropertyCell

            Boolean→checkbox · PickList→select · Date→JS datepicker · else→input

Fig 5 · Two Abstract Factories inspect the runtime DOM/PropertyType and return the right polymorphic implementation. Callers say "set this cell to this value" and never know it's a checkbox, a select2, or a flatpickr date.

Designer's mind · factories instead of if/else in the step

A CPQ cell can be text, money, boolean, picklist, rich-text, or date — each edited differently. That if/else in every step would duplicate hundreds of times. Pushing the decision into PropertyCellFactory + polymorphic subclasses means "how to edit a date cell" lives in one place and SetValueAsync is uniform. Open/Closed in action: a new cell type = a new subclass + one factory branch, zero caller changes. The crown jewel — and the steepest learning curve.

Composition over inheritance, why: parent-scoped components (ItemActionComponent(page, parentSelector)) are composed because the same button-group appears inside many containers with different parents — composition lets one class serve all of them; inheritance couldn't express "same widget, different location." ~30 modals under Shared/ModalDialogs/ each wrap one dialog and are constructed on demand (new DeleteModal(CurrentPage)).

Designer's mind · why these interfaces exist (ITableActions, IPropertyCell, IEntityTable, ISearchActions)

An interface is created here for one reason: a caller needs to depend on a capability without knowing the concrete type. TableComponent talks to IEntityTable, so it works whether the factory returned a real grid (EntityTable) or a card layout (EntityTableStandard). The factory + interface pair is what makes the polymorphism possible: the factory chooses the concrete class, the interface hides which one it chose. Contrast the alternative — a concrete EntityTable everywhere — which would force if (isGrid) … else … at every call site. Interfaces here are small and role-focused (Interface Segregation): ISearchActions is only search, IPropertyCell is only get/set — no god-interface. Rule of thumb: introduce an interface when you have (or will have) more than one implementation the caller shouldn't distinguish — not speculatively for a single class.

Section 18

API framework

A hand-rolled REST client on HttpClient + Newtonsoft. One base class owns HTTP mechanics; per-entity services express the domain.

HTTP client

Raw System.Net.Http.HttpClient — a single static shared instance in ApiService.

Serialization

Newtonsoft (JsonConvert, NullValueHandling.Ignore, JObject patching).

Error handling

EnsureSuccessStatusCode() on every call; DeleteAsync returns status instead of throwing.

Retry / logging

No retry policy in the main layer; logging is via NUnit output/Xray log files, not a logging package.

The request pipeline & HttpClient lifecycle

 ApiSteps ("Api, I create new Quote")  ── builds payload via Models/Helpers
   ─► QuoteService : ApiService         ── domain verbs (CreateQuote, CopyQuote, GetByFilter)
        ─► PostAsJsonAsync<TIn,TOut>    ── serialize → StringContent(application/json)
             ─► AddRequestAuthorizationHeader  ── token from ApiAuthentication (cached on org)
                  ─► static HttpClient.SendAsync
                       ─► EnsureSuccessStatusCode ─► DeserializeResponse<T> (stream + JsonSerializer)

protected async Task<TOut> PostAsJsonAsync<TIn,TOut>(string url, TIn body) {
    var req = new HttpRequestMessage(HttpMethod.Post, url) {
        Content = new StringContent(JsonConvert.SerializeObject(body), Encoding.UTF8, "application/json") };
    await AddRequestAuthorizationHeader(req);
    var res = await s_client.SendAsync(req);   // s_client is a single static instance
    res.EnsureSuccessStatusCode();
    return DeserializeResponse<TOut>(res);
}
Why a static HttpClient
Straight from MS guidance: a new HttpClient per request exhausts sockets under load. One shared static instance reuses connections — correct for a suite firing many requests across 10 workers.
Why HttpClient, not RestSharp/Refit
Zero extra dependency, full control over the change-envelope shapes (bulk update), and easy JObject patching. Refit would be terser for static endpoints but fights dynamic payloads.
Verb wrappers
PostAsJsonAsync, GetAsync<T>, PutAsJsonAsync, PatchAsJsonAsync, PutRawAsync, DeleteAsync, BulkUpdateAsync.
Designer's mind · Template Method at the service layer

ApiService owns how to send (serialize, auth, deserialize, error-check); each *Service owns what to send (endpoints, shapes). A new endpoint is a subclass method reading like the domain — CreateQuoteFromBasket(basketId) — with all HTTP tedium inherited. ApiServiceFactory (static) centralises construction. This keeps the API layer thin: the boilerplate is written once.

Good practice · BulkUpdate mirrors the browser

BulkUpdateAsync PUTs to /v3/api/bulk/update with the same change-envelope the UI sends, so server-side pricing recalculation runs exactly as for a user. Plain entity PUTs skip that logic — choosing bulk-update for PA changes keeps test data correctly priced.

Trade-off · no retry / no structured logging

The main layer relies on EnsureSuccessStatusCode with no retry — a transient 5xx fails the setup. If setup flakiness appears, wrap SendAsync in a small retry (e.g. Polly, or a hand-rolled loop) at the ApiService level so every service benefits at once. Same for logging: add it in the base, not per service.

Authentication — deep

Token lifecycle, headers, and the security considerations behind the API auth.

// CpqRestApi/RestHelper/ApiAuthentication.cs
public async Task<string> GetAuthToken(Organization org, HttpClient client) {
    if (string.IsNullOrEmpty(org.AuthToken)) {         // cached ON the org object
        var body = new AuthTokenRequest {
            client_id = "SpecflowUiTests", grant_type = "password",
            username = org.UserName, password = org.Password, platform = "cpq" };
        // POST {org.FullUrl}/v3/api/oauth2/token  → EnsureSuccessStatusCode → AuthToken
        org.AuthToken = token.access_token;
    }
    return org.AuthToken;
}
// header: httpRequest.Headers.TryAddWithoutValidation("Authorization", token);  // NB: no "Bearer " prefix
Token lifecycle
OAuth password grant; the access_token is cached on Organization.AuthToken and reused for the whole scenario. In-memory only.
Refresh
None — there's no expiry check. Fine for short scenarios; a long one could hit expiry.
Headers / cookies
Raw token set directly as the Authorization value (no Bearer prefix — that's what this API expects); Accept: application/json set in the ApiService ctor. No cookie handling (the API is token-based; the UI session is separate, driven by Playwright).
Security considerations
Credentials come from orgsettings.json — which currently commits real passwords (see §06 watch-out). The right shape is already here (per-org creds, env overridable); the fix is moving secrets out of source control.
Section 20

Converting a cURL command into framework code

The everyday workflow: you have a raw cURL (from browser DevTools or a colleague) and need it as a reusable, authenticated, tested part of the framework. Here is the full path.

  1. Understand the cURL. Break it into its parts — method, URL/path, headers, and body. Example to work through: 
    curl 'https://cpqstage-autotestspecflowui.vendavocpq.com/v3/api/workflows/copyquote/123' \
  -X GET \
  -H 'Authorization: eyJhbGci...' \
  -H 'Accept: application/json'
    Identify: verb GET, path /v3/api/workflows/copyquote/{id}, auth header, no body.
  2. Import into Postman (Import → Raw text → paste cURL). Postman parses it into method/URL/headers/body so you can send it, tweak params, and confirm the real response shape before writing any C#.
  3. Understand the headers. Decide which are framework-managed vs request-specific. Here Authorization and Accept are already handled by ApiService (AddRequestAuthorizationHeader + the Accept ctor header) — so you drop them from your code; the base class adds them. You only keep genuinely per-call headers.
  4. Understand the payload. For a POST/PUT, capture the JSON body from Postman and map it to a model. If a matching entity exists in GuidedSelling.Web.Entities.dll or Models/, reuse it; otherwise add a DTO under Models/DTO or Models/CPQEntity. (Our example has no body.)
  5. Generate C# from the cURL (Postman → Code snippet → C# / RestSharp, or DevTools → Copy as → C#). Treat the output as disposable scaffolding — it will create its own HttpClient, hard-code the URL and token, and inline the JSON. Never commit this; it violates every rule of the framework.
  6. Convert to production quality — strip everything the framework already owns: 
     generated (throwaway)                    framework (production)
 new HttpClient()               ───►      inherited static s_client in ApiService
 hard-coded URL + token         ───►      org.FullUrl + AddRequestAuthorizationHeader
 raw SendAsync + ReadAsString   ───►      GetAsync<T> / PostAsJsonAsync<TIn,TOut>
 manual JsonConvert             ───►      DeserializeResponse<T> in the base
 no error check                 ───►      EnsureSuccessStatusCode (inherited)
  7. Refactor into the existing service — add a domain method to the right *Service (create a new XxxService : ApiService only if no entity fits). For our example, it belongs on QuoteService: 
    // CpqRestApi/RestHelper/QuoteService.cs
public async Task<Quote> CopyQuote(string quoteId) =>
    await GetAsync<Quote>($"/v3/api/workflows/copyquote/{quoteId}");   // auth+deserialize inherited
  8. Create reusable methods, not one-offs. Parameterise the volatile parts (the id), name the method for the domain intent (CopyQuote, not Get123), and let the base handle transport. If the endpoint needs server-side recalculation, use BulkUpdateAsync instead of a plain PUT.
  9. Integrate authentication — nothing to do: because the method routes through ApiService, the per-org cached token is injected automatically. This is the payoff of the Template Method base — new endpoints are authenticated for free.
  10. Expose it to BDD & test it. Add a binding in PageObjects/ApiSteps/QuoteApiSteps.cs, register the result for cleanup, and store it as the current entity: 
    [When(@"Api, I copy Quote with Number ""([^""]*)""")]
public async Task WhenICopyQuote(string number) {
    var source = await QuoteService.GetQuoteByFilter("QuoteNumber", number);
    var copy   = await QuoteService.CopyQuote(source.QuoteId);
    AddCreatedEntity(new CreatedEntity(copy.QuoteId, "Quote"));   // cleanup will delete it
    ScenarioContextProperties.CurrentQuote = copy;                // later steps/placeholders use it
}
    Then drive it from a scenario: Given Api, I copy Quote with Number "CurrentQuoteNumber".
Designer's mind · why the framework makes cURL conversion trivial

Notice steps 6–9 are almost entirely deletion. Because ApiService centralises transport, auth, and deserialization, converting a cURL collapses to "pick the service, add one typed method." The whole point of the base class is that the interesting part of a new endpoint is one line of domain intent — everything else is inherited. A framework earns its keep when adding capability means writing less, not more.

Cleanup pipeline

Whatever a test creates, it removes — reliably, in isolation, and without one failure blocking the rest.

 During scenario: AddCreatedEntity(new CreatedEntity(id,"Quote")) → ScenarioContextProperties.CreatedEntities (locked, deduped)

   1 ApplicationSettingsCleanup · 2 ProfilePropertyCleanup · 3 EntityCleanup · 4 UserCleanupAction

Designer's mind · Command pattern + fail-soft teardown

Each cleanup is an ITestCleanupAction (a Command). The hook just iterates — a new cleanup concern is a new class, not a hook edit (Open/Closed). Add a new scenario — the full 12-point decision walkthrough

Tomorrow a new test case lands. This is the exact engineering process, one decision per line:

  1. Analyze the requirement. Name the screen(s) touched, the precondition data, the single behaviour under test, and the observable outcome. That four-part read decides everything below.
  2. Reuse an existing feature file, or create a new one? Same area and same page family → append to the existing .feature. New area/screen family → new Area_SubArea_Group.feature in the matching CoreCpqTests/ folder. Rule of thumb: a feature file = one screen family's behaviours.
  3. Write the scenario in the house grammar (PageName, I …), with @REQ_CPD on the feature and @TEST_CPD + suite tags on the scenario. Seed expensive preconditions with Given Api, I …, not UI clicks.
  4. Identify reusable steps. Read the scenario line by line and, for each, search existing *Steps.cs for a matching phrasing/regex before writing anything.
  5. When to reuse an existing step definition: if a step's regex already matches your sentence, reuse it verbatim — that's free reuse and the whole point of the grammar. Adjust your wording to fit an existing step rather than cloning it.
  6. When to create a new step definition: only when no regex matches and the action is genuinely new. Add it to the page's *Steps.cs, keep it thin (one page/service call + UpdateCurrentPage()), never logic.
  7. When to reuse an existing page-object method: if the screen already exposes the action (a ClickSave, a table setter), call it. The page is the reuse boundary for UI mechanics.
  8. When to create a new page-object method: when the screen gains a new interaction. Add it to the relevant *Page.cs (or a shared component / PropertyCell subclass if it's grid/modal behaviour), with its selectors in the nested Selectors class.
  9. Where to update ScenarioContext: in the step that creates/loads an entity — set ScenarioContextProperties.CurrentQuote (etc.) so later steps and placeholder tokens resolve it. Never store raw values under ad-hoc keys.
  10. Where assertions go: in Then steps via .Should() (AwesomeAssertions). Read state through the page/service or ScenarioContextProperties; keep the assertion in the step, the "how to read" in the page.
  11. How to avoid duplicate code: reuse steps → reuse page methods → reuse components. If you're about to copy a block, it belongs on a page/component/service instead. Register created entities with AddCreatedEntity so you never hand-write teardown.
  12. How to follow existing conventions: match the tag set, the PageName, prefix, the "no blank line before Examples:" rule, Scenario Outline only for ≥2 real variants, and English-only text (per CLAUDE.md).

Add a new page

  1. Create XxxPage.cs in PageObjects/PageObjects/[Feature]/, inherit PageObjectBase, call VerifyPageObjectUrlAndState(nameof(XxxPage)).
  2. Add nested static class Selectors (const CSS + parameterised methods); prefer IDs, then data-*.
  3. Actions as async Task<NextPage> returning the next page; settle with ClickAndWaitForLoadingAsync.
  4. Register in the UpdateCurrentPage() switch so factory-driven steps resolve it.
  5. Create XxxSteps.cs beside it; cast CurrentPage, delegate.

Add a new API endpoint/service

  1. Add a method to the right *Service, or create XxxService : ApiService (set EntityName) if no entity fits.
  2. Express the verb with inherited wrappers; use BulkUpdateAsync when recalculation must run.
  3. Add CreateXxxService(org) to ApiServiceFactory; payloads in Models/Helpers, DTOs in Models/DTO.
  4. Add XxxApiSteps.cs in ApiSteps/; on create call AddCreatedEntity and store on ScenarioContextProperties.

Add a customer override

  1. Subclass the base page in TestCustomObjects/[Customer]/PageObjects/; override the virtual selector seam or add customer-only actions.
  2. Add [Scope(Feature="…")] steps with a customer-prefixed grammar.
  3. Add the org to orgsettings.json with its own profiles. Core is untouched.
Section 27

Build it from scratch — the order

Rebuilding from an empty solution, this is the sequence and why each step must precede the next.

  1. Solution skeleton — five projects, dependency direction fixed (Models is the leaf). Boundaries are expensive to move later.
  2. Configuration — Microsoft.Extensions.Configuration + JSON + env vars. Everything downstream needs a URL and creds.
  3. Domain models — reference/author entity shapes. API and pages both bind to these.
  4. Browser lifecycle — centralise Playwright creation + the settle helper. Nothing UI works until launch/wait are solid.
  5. DI + hooks — wire BoDi and TestHooks; create browser in BeforeScenario, dispose in AfterScenario.
  6. ScenarioContext facade — StepsBase + ScenarioContextProperties + ContextKeys. Define state flow before steps depend on it.
  7. API layer — ApiService base + auth + one service. Data setup must exist before UI tests.
  8. Page Objects — PageObjectBase + self-validating ctor + Login page + the UpdateCurrentPage factory.
  9. Shared components — grid factories + modals, once ≥2 pages share a table. Don't abstract before the second use.
  10. Parallel execution — enable Parallelizable, add per-worker profiles, quarantine contended tests early.
  11. Reporting — LivingDoc, screenshots, then Xray/Jira as side effects.
  12. CI/CD — restore → transform config → build → install browsers → test (parallel) → publish.
Designer's mind · the ordering principle

The through-line: each layer can only be built once the state it depends on has an owner. Config owns environment → models own shape → browser owns session → context owns scenario state → pages/services own behaviour → reporting observes → CI orchestrates. Build in dependency order and every layer has solid ground; build out of order and you'll mock what you should have built.

Section 29

Trade-offs & watch-outs

Watch-out · secrets in plaintext

Xray secret, Jira token, org passwords in tracked JSON. Move to pipeline secrets / key vault (env-var layer already supports it). Top priority.

Trade-off · two REST stacks

CpqRestApi (current) vs. Core/CpqApi (legacy). Use the former; schedule the latter for deletion.

Trade-off · URL-switch page factory

Readable now; refactor to a registry when the switch outgrows a screen.

Trade-off · two assertion libraries

AwesomeAssertions + FluentAssertions; converge on one.

Trade-off · no API retry / token refresh

Add both at the ApiService base if setup flakiness or long scenarios appear.

Bet that paid off · the settle helper

One WaitForLoadingAsync is why the suite is stable at 10× parallel. Keep it central.

Bet that paid off · isolation by construction

Per-scenario browser + per-worker profile + scenario-scoped context ⇒ parallelism safe by default.

Exercises to cement it

  1. Trace one scenario on paper: for each line name the step, page, component, and what's written to ScenarioContextProperties.
  2. Add a new page + one action + one step end-to-end, including the UpdateCurrentPage switch entry.
  3. Convert a real cURL from DevTools into a new *Service method + ApiStep, authenticated and cleaned up (follow §20 exactly).
  4. Add a new ITestCleanupAction and watch it run without editing the hook — feel Open/Closed.
  5. Add a PropertyCell subclass for a fictional cell type and wire it into PropertyCellFactory — the hardest, most instructive change.
  6. Introduce a deliberately ambiguous step binding and read Reqnroll's error — internalise the exactly-one-match rule from the failure.

The one-paragraph takeaway

This framework is a disciplined answer to one question — who owns which state, and for how long? Config owns the environment (run-lived); the browser and ScenarioContextProperties own the test (scenario-lived); page objects own the view (view-lived). Every pattern — DI restraint, the settle helper, the context facade, the entity-grid factories, the ApiService template method, fail-soft cleanup, side-effect reporting, quarantined non-parallel tests — exists to manage those lifetimes safely under parallel load. Learn to think in lifetimes and ownership and you can not only maintain this framework — you can build the next one.
