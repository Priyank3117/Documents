Part 0 · Mindset
Section 0

How an expert (or AI agent) actually thinks

Seniority is not memorised syntax or typing speed. It is a loop run without being told to, and a refusal to act on anything unconfirmed. Everything else in this playbook is a tool that loop wields.

Expert's mind — the reasoning behind a move Good habit to build Trade-off / judgement call Trap that wastes hours

0.1 The core loop: recon → plan → implement → verify → summarize

Every competent change follows the same five beats. Each beat exists to kill one category of wasted time; skipping a beat is exactly where debugging hours disappear.

   ┌─────────┐   ┌────────┐   ┌───────────┐   ┌────────┐   ┌───────────┐
   │ 1 RECON │──►│ 2 PLAN │──►│3 IMPLEMENT│──►│4 VERIFY│──►│5 SUMMARIZE│
   └─────────┘   └────────┘   └───────────┘   └────────┘   └───────────┘
   read facts    what/where    smallest        narrow→wide   what changed,
   before        /how-verify   idiomatic        revert-test  why, residual
   touching      (write it)    change           build+lint   risk
        ▲                                                          │
        └──────────────── new facts feed the next loop ◄───────────┘
The five beats and the specific failure each one prevents.
BeatWhat you actually doWaste it prevents
1 · ReconRead structure, entry points, conventions, the failing test. Gather facts.Editing the wrong file; rebuilding something that exists.
2 · PlanState what changes, in which files, in what order, and how you'll verify.Half-finished edits; forgotten callers; scope creep.
3 · ImplementSmallest change that satisfies the plan, in the local style.Big speculative rewrites nobody can review or revert.
4 · VerifyRun the narrowest test, widen; read output; check callers & build."It compiles" shipped as "it works"; silent regressions.
5 · SummarizeRecord what changed, why, what's verified, residual risk.Lost context; re-deriving everything tomorrow.
Expert's mind · why it's front-loaded

Most debugging time is not spent fixing — it is spent fixing the wrong thing because recon was skipped. Ten minutes of recon routinely saves hours of blind editing. An AI agent is disciplined about this because it cannot afford wrong guesses across many tool calls; a human should be disciplined for the same reason.

0.2 "Guess and edit" vs "confirm and edit"

These are two whole operating modes. The entire playbook exists to move you permanently into the second.

SituationGuess & edit (amateur)Confirm & edit (expert)
"This method looks unused."Delete it.rg "MethodName" across src + tests + config, then decide.
"The bug is in this method."Change lines to see what helps.Reproduce in one test, read the trace, confirm the exact line.
"This timeout should be 30s."Change it and hope.Find where it's read and who else depends on it, then change.
"Tests pass — done."Push.Confirm the change is exercised (revert-test), run callers, smoke.
Expert's mind · a distinct example

A test asserts a quote total of 110 but gets 100. The guesser edits the assertion to 100 and moves on — hiding a real pricing bug. The confirmer traces the 100 back to a discount applied twice, fixes the source, and the assertion passes honestly. Same red test; opposite outcomes. The only difference was confirming the cause before editing.

0.3 Why narrating the plan first catches most mistakes

Saying the plan out loud — even to yourself, even in a scratch comment — forces the vague into the specific, and specifics expose flaws that fuzzy intentions hide.

  1. It surfaces unknowns. "I'll update the discount logic" collapses the moment you try to name the file — you realise you don't yet know where it lives. Discover that now, not mid-edit.
  2. It reveals missing consumers. Writing "change CalculateTotal()" prompts "who calls it?" — which you answer before breaking three callers.
  3. It sets a stopping condition. A plan ending in "verified by dotnet test --filter ~QuoteTotal" tells you exactly when you're done.
Good habit · the three-line plan

Before any change, write three lines in a scratch file: WHAT I'm changing, WHERE (files), HOW I'll verify. If you can't fill all three, you're still in recon — stay there. This is the single highest-leverage habit in the book.

0.4 What to copy from how an AI agent works

  • It searches before it reads (grep/glob) and reads before it writes — never edits a file it hasn't opened.
  • It reads the whole relevant region, not one line, so the edit matches surrounding style automatically.
  • It runs the narrowest verification that proves the change, then widens.
  • It states assumptions explicitly, so a wrong one is visible and cheap to fix.

You have the same tools it does — rg, git, dotnet test, a stack trace. Parts 1–9 are those tools; Part 0 is the loop that wields them.

Part 1 · Recon & Navigation
Section 1

Reading a .NET repository fast

Your first pass builds a map, not full understanding: where things are, how it builds, how it tests, and which conventions rule. Detail comes on demand once the map is right.

1.1 The first-10-minutes checklist

  1. Read README / CONTRIBUTING — how to build, test, run. Two minutes saves twenty.
  2. List the top level — Get-ChildItem / ls. Identify src vs tests vs config vs docs vs CI.
  3. Open the .sln — it names every project and how they group.
  4. Read the CI file (azure-pipelines.yml, .github/workflows/*.yml) — the authoritative build+test recipe, because it's what actually runs green.
  5. Find the test projects — folder names + the test framework tell you how the team verifies.
  6. Find config/env files — appsettings.json, *.runsettings, .env — they reveal external dependencies (URLs, DBs, creds).
  7. git log --oneline -20 — what's changing and the commit-message style.
  8. Build once: dotnet build — a green build confirms your environment before you touch anything.
  9. Run one test — confirms the test path works and shows the runner's output format.
  10. Note conventions — naming, folders, formatting, .editorconfig. You will match these, not fight them.
Expert's mind · why CI beats the README

READMEs drift; CI does not. If the pipeline is green, its commands are correct today. When docs and CI disagree, believe CI — copy its exact dotnet test line and run that.

1.2 The solution & project model (.sln / .csproj)

A .NET repo is a solution (.sln) containing projects (.csproj). Each project compiles to one assembly (DLL/EXE) and declares its own dependencies. Understanding the project graph is understanding the architecture.

# What SDKs/runtimes are installed, and which the repo resolves to
dotnet --info
dotnet --list-sdks

# The solution's projects and their relationships
dotnet sln list                          # every project in the .sln
dotnet list SpecFlowTests/SpecFlowTests.csproj reference   # project-to-project deps
dotnet list SpecFlowTests/SpecFlowTests.csproj package     # NuGet deps + versions

# Prove the environment is good before editing anything
dotnet restore
dotnet build -c Release

ProjectReference edges are one-directional and enforced by the compiler: if Models does not reference PageObjects, then no domain model can accidentally depend on a page object. Reading the reference graph (dotnet list … reference on each project) tells you the intended layering before you read a single class — and tells you instantly where a new piece of code is allowed to live.

Part 2 · Recon & Navigation
Section 2

Finding linkages between files

Code is a graph, not a list. Before you touch a node you must see its edges — who calls it, what it calls, when it last changed and why. These techniques turn an opaque solution into a navigable map.

2.1 The universal layer: git + ripgrep (the 80%)

You can trace almost anything with git and a fast grep, with no IDE. Learn these cold — they work even over SSH on a build agent.

# Find every reference to a symbol (ripgrep respects .gitignore, is fast)
rg "CalculateTotal"                 # all hits, file:line
rg -w "Total"                       # whole word (won't match SubTotal)
rg -t cs "CalculateTotal"           # C# files only
rg -n -C 3 "CalculateTotal"         # 3 lines of context each side
rg -l "CalculateTotal"              # just the file names
rg -g "!**/bin/**" -g "!**/obj/**" "CalculateTotal"   # exclude build output

# Who changed a line, and in which commit?
git blame -L 40,60 src/Pricing.cs

# The behavioural history of a file
git log -p -- src/Pricing.cs        # every diff that touched it
git log --follow -- src/Pricing.cs  # keep history across RENAMES
git log -S"CalculateTotal"          # commits that ADDED/REMOVED that string (pickaxe)
git log -G"discount.*0\.1"          # commits whose diff matches a regex
git show <commit>                    # the full change + message (the "why")
Expert's mind · the define / consume / test triangle

For any symbol, answer three questions with the commands above: (1) Definition — where declared? (2) Consumers — who references it (rg -w)? (3) Tests — which test names mention it? Sketch that triangle on paper. It is the unit of safe change: you may edit the definition only once you can see every consumer and the tests that guard them.

Trade-off · rg finds text, not semantics

rg can't tell an overload apart, resolve an interface to its implementations, or follow reflection. It's the fast first pass. For overloads, inheritance, and interface implementations use Roslyn (2.2). And always rg the string name too, because C# can bind by string — DI registration, reflection, config keys — which the compiler won't catch on a rename.

Roslyn is the C# compiler-as-a-service; every "smart" navigation in Visual Studio and Rider is built on it. Use the IDE for semantics, the CLI for repeatability.

When a test binds to a referenced DLL you don't have source for (like a domain-entities assembly), decompile it with ilspycmd to read the real property names, types, and shapes instead of guessing. Guessing at an entity's property name is the #1 cause of silent test-data bugs.

2.3 Worked example — trace a failing UI test to its root, in C#

Scenario. A Reqnroll UI test fails asserting an order total. Trace from the failing scenario → the step → the page/service method → the logic → back to the test that guards it.

 .feature line            Step [Binding]           Page/Service           Domain logic
 "Basket, I verify   ──►  BasketPageSteps      ──► BasketPage        ──►  CalculateTotal()
  the total is 110"        .ThenTotalIs(...)        .GetTotalAsync()        (the actual bug)
        │                       │                        │                      │
        └─── rg the phrase ─────┴──── rg the method ─────┴──── go to def ───────┘

# 1. Find the failing scenario / assertion phrase
rg -t cs -n "total is" ; rg -g "*.feature" "the total"
# 2. Find the step binding that matches that Gherkin line
rg -t cs -n "the total is"          # lands in BasketPageSteps.cs [Then(...)]
# 3. Find the page/service method the step calls
rg -t cs -n "GetTotalAsync|CalculateTotal"
# 4. See when the logic last changed and why
git log -p -S"CalculateTotal" -- src/
# 5. Reproduce just that one test (fast loop)
dotnet test --filter "FullyQualifiedName~OrderTotal"
Expert's mind · trace along the layers, not randomly

Notice the trace follows the framework's own layering: feature → step → page → logic. Because an automation framework is layered on purpose, tracing is deterministic: each layer names the next. If you find yourself grepping randomly, you've lost the layer thread — go back to the last file you understood and read which layer it calls into.

2.4 Building the mental dependency map

  • Top-down: from the .sln, list projects and their ProjectReference edges → the architecture skeleton.
  • Middle-out: pick the class you're changing; find its consumers (Find All References) and its dependencies (what it news / injects).
  • Bottom-up (history): git log --follow -p on the file to learn why it is the way it is before "improving" it.

2.5 The AI agent's glance-and-map arsenal

This is how an agent like Claude Code — and a fast human — "glances" through an unfamiliar codebase and reconstructs relationships in seconds, without opening every file. The trick is to ask the repo structured questions with search, instead of reading linearly.

Step 1 · Get the shape of the repo instantly

# Every tracked file — the fastest possible overview (no build output noise)
git ls-files
git ls-files "*.cs" | rg -v "/(bin|obj)/" | wc -l      # how big is the C# surface?
git ls-files "*.csproj"                                  # the project list = the architecture
git ls-files "*.feature" | head                          # the behaviours under test

# List files by name super-fast (ripgrep can enumerate, not just search)
rg --files -t cs | rg -i "page|steps|service"            # the POM / BDD / API surface at a glance
fd -e cs -E bin -E obj                                    # fd: same idea, excludes build dirs

# Directory tree without the noise (PowerShell / cross-platform tree)
git ls-files | %{ ($_ -split '/')[0..1] -join '/' } | Sort-Object -Unique   # top 2 levels

Relationships in C# are expressed as inheritance, interface implementation, instantiation, injection, and attribute use. Each is a searchable pattern:

Expert's mind · why an agent searches instead of reads

Reading a 300-file repo top to bottom is O(n) and forgets the start by the end. Searching is O(1) per question: "what implements IEntityTable?" is one command, answered in milliseconds, and the answer is the relationship. An agent builds the map by asking a dozen precise questions, not by reading a dozen files. Copy that: form the question, express it as a search, read only the 3–5 lines that answer it.

Step 3 · Trace history & plan the change from git

Before changing a function, read its life story — git can trace a single function across its whole history:

# Follow ONE method's evolution across the whole history (super powerful)
git log -L :CalculateTotal:src/BasketBasePage.cs   # every commit that changed that method's body
git log -L 40,60:src/Pricing.cs                    # same, by line range

# Who owns / knows this area (for questions & review)?
git shortlog -sn -- src/Pricing.cs                 # commit counts per author for this file
git blame -w -C -C src/Pricing.cs                  # -w ignore whitespace, -C detect moved/copied code

# What changed recently & how big — scope a task at a glance
git log --oneline --stat -5                        # recent commits with file churn
git diff --stat main...HEAD                         # size of your branch vs main
git log --oneline --graph --decorate --all         # the branch topology

# git grep — search AT ANY commit/branch (search the past, not just now)
git grep -n "CalculateTotal" <commit>              # where was it before a refactor?
git grep -l "IEntityTable"                          # files (tracked) referencing it
Good habit · git log -L is the planning superpower

Before editing a method, run git log -L :Method:File. In one command you see every change ever made to that method, with messages — the bugs it already had, the edge cases previously handled, and who to ask. You plan the change knowing the history instead of re-introducing an old bug.

Step 4 · From map to plan

  • Enumerate the surface (git ls-files, rg --files) → know what exists.
  • Query the relationships (the table above) → know how the target connects.
  • Read the history (git log -L, blame -C) → know why it's shaped this way.
  • List the consumers (rg -w + string search) → know the blast radius (Part 3).
  • Now write the three-line plan (Part 0.3). Only now do you touch code.

Section 3

Planning & scoping work before touching code

A change you can't describe in advance is a change you don't understand yet. Planning turns a vague task into a bounded, verifiable sequence — and tells you the risk before you incur it.

3.1 Classify by blast radius

The first decision is how far the change reaches. It sets how much recon and verification you owe.

ClassExample (C# automation)Recon owedVerification owed
Single-lineFix a locator string; bump one timeoutConfirm nothing else reads the exact valueBuild + the one affected test
Single-fileAdd a method to one page objectRead the whole page; match its styleTests that exercise that page
Cross-cuttingChange a shared PageObjectBase method, or a step used by many features, or an API service signatureEnumerate every caller (Part 2), including string/reflection callersAll callers' tests + full dotnet build + smoke
The classic mistake

Treating a cross-cutting change as single-file. Editing a shared helper "just for my case" and shipping when your one test passes — while three other callers now silently misbehave. If a symbol has many consumers, it is never a single-file change. In a POM/BDD framework the shared bases (PageObjectBase, StepsBase, an API ApiService) are the highest-blast-radius files in the repo — treat every edit to them as cross-cutting.

3.2 The plan template

Fill this in before editing. It fits in a scratch file or the PR description and is the artefact you reason from.

GOAL:      one sentence — the observable outcome when done.
CHANGES:   file-by-file, in the order you'll make them.
           1. BasketBasePage.cs → fix double-applied discount in CalculateTotal
           2. BasketTotalTests.cs → add a regression test for the discount case
CONSUMERS: everyone who calls the changed code (from Part 2 search).
           - BasketPageSteps.cs, QuoteDetailPage.cs (checked: signature unchanged)
VERIFY:    the exact commands that prove success.
           - dotnet test --filter "FullyQualifiedName~BasketTotal"
           - dotnet build   (no new warnings)
RISK:      what could still be wrong, and why it's acceptable.
           - Multi-currency rounding untouched; out of scope, tracked separately.

3.3 Identify every caller before you change

  1. List consumers: rg -w "SymbolName", then Find All References (Roslyn) to catch overloads accurately.
  2. Also grep the string name — DI (services.AddScoped<X>()), reflection, config keys, and Gherkin step regexes bind by name and won't show as compile errors on a rename.
  3. Separate reads from writes: who merely calls it vs who depends on its exact return shape / side effects.
  4. Note each consumer's tests — those are your safety net; list which must stay green.
  5. Decide compatibility: keep the signature (safe, additive overload) or break it (then update all callers in the same change).
Expert's mind · the search that changes the design

You want GetUser(int id) to also accept an email. Guess-and-edit adds an overload and moves on. Confirm-and-edit runs rg -w "GetUser", finds 14 call sites and 6 tests, notices three call it by name via reflection, and therefore chooses an additive overload — precisely because a signature change would break the reflection callers the compiler can't see. The search didn't just inform the change; it dictated the design.

Part 4 · Recon & Navigation
Section 4

Deciding where code belongs

Correct code in the wrong place is debt on day one. Placement decides who can reuse it, who must change when it changes, and whether the next person can find it. In a C# automation framework this is the decision engineers get wrong most often.

4.1 SOLID — the practical test you actually apply

PrincipleThe question you askC# automation example
Single Responsibility"Do I need the word and to describe this class?" If yes, split.A BasketPage that also does HTTP → split the HTTP into a BasketService.
Open/Closed"Can I add the new case without editing existing code?"New editable cell type → a new PropertyCell subclass + one factory branch, not an if in every step.
Liskov"Can the subclass stand in for the base with no surprises?"A customer ClearwaterBasketPage : BasketPage must not throw on inherited methods.
Interface Segregation"Does any implementer leave methods empty / NotImplemented?"Prefer small ITableActions, ISearchActions over one fat IPage.
Dependency Inversion"Does high-level code new up a concrete detail?"A step depends on PageObjectBase/an interface, not on new ChromeDriver() directly.

4.2 C# structural conventions

  • Namespace mirrors folder: PageObjects.Quote.Basket lives in PageObjects/Quote/Basket/. Breaking this makes types unfindable.
  • One public type per file, file named after the type (BasketPage.cs holds BasketPage).
  • Extension methods go in a static class XxxExtensions — use them to add verbs to a type you don't own (e.g. this IPage from Playwright). Don't use them to hide logic that belongs on your own class.
  • Interfaces prefixed I, kept beside their primary implementation or in an Abstractions/ folder. Create one only when there is (or will be) more than one implementation the caller shouldn't distinguish.
  • Partial classes only for generator output (e.g. Reqnroll's *.feature.cs) or a genuinely large cohesive type — never to hide a god class across files.
  • Static helpers for stateless, cross-cutting operations only (UrlHelper, a factory). A static class holding unrelated methods is a smell (4.4).
// Extension method done right: a verb on a type you don't own (Playwright's IPage)
public static class PlaywrightExtensions
{
    public static async Task ClickAndWaitAsync(this IPage page, string selector)
    {
        await page.Locator(selector).First.ClickAsync();
        await page.WaitForLoadStateAsync(LoadState.NetworkIdle);
    }
}
// Usage reads like a native Playwright call:
await page.ClickAndWaitAsync("#SaveButton");

4.4 The two anti-patterns that define junior automation code

Anti-pattern 1 · logic in step definitions

A [When] step with an if/for/calculation inside it. Steps must be one page/service call plus a context update — no branching business logic. If a step computes an expected value, that computation belongs in a builder or the page. Test: if you can't read a step method aloud as a single sentence, it's doing too much.

Anti-pattern 2 · the god "Helpers" class

A 2,000-line TestHelper / Utils of unrelated statics. Before adding to it, ask: is this behaviour of a screen (→ page), a data shape (→ builder/model), or a transport (→ service)? It almost always is — and belongs there. Utilities are justified only for genuinely cross-cutting, stateless operations, and even then name them for what they do, not "Helper" / "Manager" / "Utils".

Expert's mind · the two-question heuristic

"Who else would want this, and what makes it change?" Wanted by many screens & changes with the DOM → shared component. Wanted by one screen → that page. Changes per environment → config. Changes with a data contract → a model/builder. Answer those two and placement is usually obvious — and matches how the existing framework is already organised, which is the point.

  1. Read from the failure, not the top. Search the log for the first Failed!, error, or the first red X, then scroll up to its cause.

  3. Download artifacts — the pipeline publishes screenshots, Playwright traces, .trx, and reports. Open them before re-running blindly.
  4. Copy the exact test command from the pipeline YAML and run it locally — you want to run what CI ran (same filter, same NumberOfTestWorkers).

6.6 Reading a test-run summary

 Passed!  - Failed:     0, Passed:   312, Skipped:  4, Total:   316   ← healthy
 Failed!  - Failed:    17, Passed:   299, Skipped:  0, Total:   316   ← is it 17 bugs or 1?
  1. Counts first: passed / failed / skipped. Skipped is not passed — know why they skipped (a broken [Ignore]? a filter that excluded real tests?).
  2. Find the FIRST real failure. Later failures are often fallout — a broken [SetUp] or a failed login cascades into dozens of reds. Fix the first, re-run, re-assess.
  3. Error vs failure: an assertion failure (Expected 110 but was 100) means the code ran and produced the wrong value; an exception/error (NullReferenceException) means it couldn't run. Very different root causes (Part 7).
  4. Narrow, then widen. Reproduce the one failing test in isolation (6.1) before running the whole suite again.
Expert's mind · why narrowest-first is the biggest daily time saver

Iterating against the full suite pays the whole suite's runtime on every attempt (minutes). Iterating against the single failing test pays seconds. Only once it's green do you widen — file, then suite — to catch regressions. A 2-second loop versus a 5-minute loop, run twenty times a day, is the difference between finishing at lunch and finishing at midnight.

Section 7

Reading & handling C# stack traces

A stack trace is a signed confession: the program telling you exactly where — and often why — it failed. Reading it well turns "I have no idea" into a precise, testable hypothesis in under a minute, usually with no debugger at all.

7.1 Triage in 30 seconds — what to read, what to skip

You can diagnose most failures from the trace and the test result alone. Read in this order; ignore the rest until you need it.

  1. The exception type. The category of problem — NullReferenceException, TimeoutException, an assertion. This already halves the search space.
  2. The message. Often names the offending value: Expected 110 but found 100, no element matches selector "#submit".
  3. The FIRST frame that is YOUR code. Skip framework/library frames at the top; the first frame in a file you own is where to look.
  4. The innermost InnerException / last --- End of inner exception ---. The root is usually deeper than the exception printed first.
Expert's mind · origin vs surface

An exception surfaces where it's thrown and originates where the bad state was created. A NullReferenceException surfaces at order.Total but originates wherever order was left null. The trace shows the surface; your job is to walk down the frames to the origin. This one distinction is the skill of debugging.

Good habit · diagnose from the test result first

Before opening a debugger, read the failure message + trace and write your hypothesis. Distinguish immediately: an assertion failure means the code ran and computed the wrong value (logic bug); an exception means it couldn't run (null, timeout, missing element). That fork points you at logic vs. environment/timing before you touch anything.

7.2 General anatomy — yours vs library frames

System.NullReferenceException: Object reference not set to an instance of an object.
   at Vendor.Grid.Render(Cell c)              ← library frame (skip)
   at MyApp.Pricing.CalculateTotal()          ← ★ FIRST of MY frames — start here  Pricing.cs:41
   at MyApp.Checkout.Submit()                 ← my caller (how we got here)       Checkout.cs:88

Read a trace like the call stack it is: the top frame is where it blew up; each frame below is who called it. To understand why the top frame got bad input, read downward to the caller that supplied it. In a Release build without PDBs you may lose line numbers — build Debug or ship symbols to get File.cs:line.

7.6 Hypothesis-driven debugging

Debugging is science, not poking. Form a hypothesis, run one experiment that can disprove it, iterate. Guessing changes many things at once and learns nothing; this learns something every step.

  1. Isolate. Reduce to the smallest failing case — one test, one input: dotnet test --filter "FullyQualifiedName~TheOneTest".
  2. Reproduce reliably. If it fails only sometimes, make it fail every time (fixed data, disable retries, run serial) before trusting any fix.
  3. Read the trace & form ONE hypothesis. "The discount is applied twice in CalculateTotal." Specific and falsifiable.
  4. Instrument to confirm/kill — don't fix yet. Add one TestContext.WriteLine($"discounts={count}") or a breakpoint that will prove or disprove the hypothesis.
  5. Confirm. Run once. Match? If not, discard the hypothesis (don't patch around it) and form a new one from what you learned.
  6. Fix at the origin. Change the root cause — not the symptom, and never the assertion.
  7. Re-verify narrow → wide, remove instrumentation, confirm no new warnings.
The cardinal sin

Changing multiple things at once "to see what happens." When it goes green you won't know which change fixed it — or whether you masked the bug and introduced another. One hypothesis, one experiment, one confirmation. Every time.

Expert's mind · worked mini-example (C#)

Symptom: Expected 110 but was 100. Isolate: dotnet test --filter ~QuoteTotal. Read: it's an assertion, not an exception → code ran, logic wrong. Hypothesis: discount applied twice. Instrument: TestContext.WriteLine the discount count in CalculateTotal. Confirm: prints 2. Fix: remove the duplicate application at its source (a base method AND an override both applying it). Re-verify: passes honestly; remove the log. Minutes, because every step was aimed.

10.4 Diagnosing flaky tests — systematic elimination

A flaky test passes and fails without code changes. Never "just re-run" — that trains the team to ignore real failures. Eliminate causes in order of likelihood.

CauseSignatureFix
Bad waits (race)Fails on slow agents; "not found/interactable"Wait for explicit state; never Thread.Sleep
Shared / static stateFails in suite, passes aloneIsolate per test; beware static fields & NUnit fixture reuse
Test-data collisionsFails under parallelPer-worker data / unique IDs; no shared mutable fixtures
Animations / overlaysIntermittent click interceptedDisable animations; wait for the overlay to detach
External dependencyFails when a service is slow/downStub it, or retry only that boundary with backoff
Time / localeFails at midnight / other timezoneInject the clock; set explicit CultureInfo
  1. Reproduce it — loop the test until it fails: dotnet test --filter ~Flaky in a shell loop, or NUnit [Retry]/repeat locally. A flake you can't reproduce, you can't fix.
  2. Bisect the cause — run it alone (order?), then headless (rendering?), then serial (NumberOfTestWorkers=1 — parallel collision?), then with fixed data.
  3. Fix the root, keep the guard — replace the sleep with a state wait; loop it again to prove it's stable.
Retries hide, they don't heal

Auto-retry belongs on a genuinely external boundary. Retrying a race condition just lowers the failure rate enough to be ignored — until it fails in a demo. Fix the wait; reserve retries for true externalities.

Working smart without AI

An AI assistant is an accelerator, not a crutch. These manual checklists reproduce what a good agent does for you — so you stay sharp and effective when it's offline, restricted, or simply the wrong tool.

12.1 Manual code-review checklist (your own diff, before commit)

  • Read every changed line as if reviewing a stranger. git diff is not optional.
  • Only intended changes present — no stray files, Console.WriteLine, commented-out code, or formatting churn.
  • Matches local conventions — naming, async style, assertion library, nullable annotations.
  • Every new branch handled — null / empty / timeout / error paths, not just the happy path.
  • Names say what they do — a reader shouldn't need the method body to understand the call.
  • A test fails if I revert the fix (the revert test, Part 8.3).
  • Callers updated — if a signature/behaviour changed, all consumers (incl. string/reflection) changed too.

12.2 Manual debugging checklist

  1. Reproduce narrowly — one test, fixed input (dotnet test --filter).
  2. Read the trace — exception type → message → first frame of my code → innermost inner exception.
  3. One hypothesis — specific and falsifiable.
  4. One experiment — a log/breakpoint that confirms or kills it, without changing behaviour.
  5. Confirm, then fix the origin — never the symptom or the assertion.
  6. Re-verify narrow → wide, remove instrumentation, no new warnings.

12.3 Manual "where does this belong" checklist

  • Who else would want this? (many screens → shared component; one screen → that page)
  • What makes it change? (DOM → page; environment → config; data shape → model/builder; transport → service)
  • Is it behaviour or data? (behaviour → code layer; data → test-data/fixtures)
  • Would putting it here create a circular ProjectReference? (if so, extract a shared piece)
  • Am I about to add logic to a step or dump it in "Helpers"? (stop — re-route per Part 4)

12.4 Build & maintain a personal knowledge base

13.2 The "how to check" reflex — never trust, verify

Advanced users don't guess what a command did — they inspect it. Three habits cover almost everything.

Check what a command returned (its shape)

# PowerShell pipes OBJECTS. Get-Member shows every property/method you can use.
Get-Process chrome | Get-Member                 # what can I read off a process object?
Get-Process chrome | Select-Object Id, Name, WorkingSet, StartTime   # pick fields
Get-Process chrome | Format-List *              # dump ALL properties of the first items

Check whether it succeeded (exit status)

# PowerShell
dotnet build
$?                 # True/False: did the LAST command succeed?
$LASTEXITCODE      # the native exe's exit code (0 = success). CI gates on this.

:: cmd.exe
dotnet build
echo %ERRORLEVEL%  :: 0 = success, non-zero = failure

Check how long it took (find the slow step)

# PowerShell: time any command or block
Measure-Command { dotnet build -c Release }     # prints TotalSeconds — profile a slow build/test
Measure-Command { dotnet test --filter ~Smoke }
Expert's mind · Get-Member is how you learn PowerShell without docs

Every PowerShell command returns objects, and | Get-Member lists exactly what you can pull off them. Stuck on "how do I get the PID of the process on port 5000?" — pipe Get-NetTCPConnection into Get-Member, see the OwningProcess property, done. This turns the shell into a self-documenting API, the same way --help does for CLIs (Part 9.1).

13.3 Advanced PowerShell for engineers

The object pipeline (Where-Object, ForEach-Object, Select-Object, Sort-Object, Measure-Object, Group-Object) is what makes PowerShell powerful. Run these from the repo root.

# Find the 10 biggest source files (bloat hunting)
Get-ChildItem -Recurse -File -Filter *.cs |
  Sort-Object Length -Descending | Select-Object -First 10 Name, Length

# Files changed in the last day (what did I / CI just touch?)
Get-ChildItem -Recurse -File |
  Where-Object { $_.LastWriteTime -gt (Get-Date).AddDays(-1) } |
  Select-Object FullName, LastWriteTime

# Count tests per file quickly (Group-Object)
Select-String -Path *.cs -Pattern '\[Test\]|\[Fact\]' |
  Group-Object Path | Sort-Object Count -Descending | Select-Object Count, Name

# Follow a live log (watch a test run / app log as it writes)

# Select-String with context + capture groups (grep on steroids)

1..5 | ForEach-Object -Parallel { "checked $_" } -ThrottleLimit 4

13.4 Process & port control — the automation lifesavers

Two failures dominate flaky local automation: a hung browser/driver that locks the next run, and a port already in use so the app-under-test won't start. These commands fix both in seconds.

# --- HUNG DRIVER/BROWSER: find and kill leftovers (PowerShell) ---
Get-Process chromedriver, chrome, dotnet -ErrorAction SilentlyContinue   # what's running?
Stop-Process -Name chromedriver -Force                # kill all chromedriver
Get-Process | Where-Object { $_.Name -match 'chrome|playwright|dotnet' } | Stop-Process -Force

:: cmd.exe equivalents
tasklist | findstr /i "chromedriver chrome dotnet"
taskkill /F /IM chromedriver.exe                      :: /F force, /IM by image name

# --- PORT IN USE: what's holding port 5000? (the app won't start) ---
Get-NetTCPConnection -LocalPort 5000                  # PowerShell: shows OwningProcess (PID)
Get-Process -Id (Get-NetTCPConnection -LocalPort 5000).OwningProcess   # which app is it?
netstat -ano | Select-String ":5000"                  # PID is the last column

:: cmd.exe
netstat -ano | findstr :5000
taskkill /F /PID 12345                                 :: kill by the PID you found

# --- IS THE APP-UNDER-TEST ACTUALLY UP? (before blaming the test) ---
Test-NetConnection localhost -Port 5000               # TcpTestSucceeded: True/False
Good habit · check the environment before re-running

When a UI test fails to even start, don't re-run it — check the environment first: Is a stale chromedriver still running? Is the app's port free? Is the app reachable (Test-NetConnection)? Nine times out of ten a "flaky start" is a leftover process or a busy port, fixed by two commands — not a code change.

13.5 Smoke an API straight from the terminal

Before writing an API test, confirm the endpoint and payload from the shell — faster than a full test run, and it validates your understanding (ties to converting a cURL, framework-handbook style).
